# Java 反射与注解深度解析

## 一、概述

反射（Reflection）和注解（Annotation）是 Java 的高级特性，也是各种框架（Spring、Retrofit、ButterKnife）的实现基础。理解它们对于阅读源码、设计通用框架至关重要。

**核心要点：**

- **反射**：运行时获取类的结构和调用方法，是实现动态代理、依赖注入的基础
- **注解**：元数据标记，配合反射或 APT 实现代码生成、配置简化
- **APT**：编译时注解处理，性能优于运行时反射，是现代框架的主流方案
- **KSP**：Kotlin Symbol Processing，Kotlin 版的 APT，速度更快

---

## 二、反射机制

### 2.1 什么是反射

反射允许程序在运行时获取任意类的内部信息，并操作其字段、方法和构造函数。

**Class 对象是反射的入口**：

```java
// 获取 Class 对象的三种方式
Class<?> clazz1 = String.class;                          // 类字面量
Class<?> clazz2 = "hello".getClass();                    // 对象的 getClass()
Class<?> clazz3 = Class.forName("java.lang.String");     // 全限定名（可能抛异常）
```

### 2.2 反射的核心 API

#### 2.2.1 获取类结构

```java
Class<?> clazz = User.class;

// 类基本信息
String name = clazz.getName();           // 全限定名：com.example.User
String simpleName = clazz.getSimpleName(); // 简单名：User
Class<?> superClass = clazz.getSuperclass(); // 父类
Class<?>[] interfaces = clazz.getInterfaces(); // 实现的接口
int modifiers = clazz.getModifiers();    // 修饰符

// 获取字段
Field[] fields = clazz.getFields();          // public 字段（含继承）
Field[] declaredFields = clazz.getDeclaredFields();  // 所有声明字段（不含继承）
Field nameField = clazz.getDeclaredField("name");    // 指定字段

// 获取方法
Method[] methods = clazz.getMethods();       // public 方法（含继承）
Method[] declaredMethods = clazz.getDeclaredMethods(); // 所有声明方法
Method setMethod = clazz.getDeclaredMethod("setName", String.class);

// 获取构造函数
Constructor<?>[] constructors = clazz.getConstructors();
Constructor<?> constructor = clazz.getDeclaredConstructor(String.class, int.class);
```

#### 2.2.2 操作字段

```java
public class User {
    private String name;
    public int age;
}

Class<?> clazz = User.class;
User user = new User();

// 获取字段
Field nameField = clazz.getDeclaredField("name");
Field ageField = clazz.getField("age");

// 设置可访问（突破 private 限制）
nameField.setAccessible(true);

// 读写字段值
nameField.set(user, "张三");
ageField.set(user, 25);

String name = (String) nameField.get(user);  // "张三"
int age = (int) ageField.get(user);          // 25
```

#### 2.2.3 调用方法

```java
public class User {
    private String name;

    public String getName() { return name; }

    private void setName(String name) { this.name = name; }

    public static void staticMethod() { System.out.println("static"); }
}

Class<?> clazz = User.class;
User user = new User();

// 调用实例方法
Method setName = clazz.getDeclaredMethod("setName", String.class);
setName.setAccessible(true);  // private 方法需要设置
setName.invoke(user, "李四");  // 调用：user.setName("李四")

Method getName = clazz.getMethod("getName");
String name = (String) getName.invoke(user);  // 调用：user.getName()

// 调用静态方法
Method staticMethod = clazz.getMethod("staticMethod");
staticMethod.invoke(null);  // 静态方法，obj 参数为 null
```

#### 2.2.4 创建实例

```java
Class<?> clazz = User.class;

// 方式一：无参构造
User user1 = (User) clazz.newInstance();  // 已废弃，只能调用 public 无参构造

// 方式二：通过 Constructor
Constructor<?> constructor = clazz.getDeclaredConstructor(String.class);
constructor.setAccessible(true);
User user2 = (User) constructor.newInstance("王五");
```

### 2.3 反射的性能问题

**反射为什么慢？**

| 原因 | 说明 |
|------|------|
| 方法查找 | 运行时查找方法，需遍历类继承链 |
| 参数装箱 | 基本类型参数需要装箱 |
| 安全检查 | 每次调用都需要访问权限检查 |
| JIT 优化受限 | 反射调用难以被 JIT 内联优化 |

**优化方案**：

```java
// 1. 缓存 Method/Field 对象（避免重复查找）
private static final Method SET_NAME_METHOD;

static {
    try {
        SET_NAME_METHOD = User.class.getDeclaredMethod("setName", String.class);
        SET_NAME_METHOD.setAccessible(true);
    } catch (NoSuchMethodException e) {
        throw new RuntimeException(e);
    }
}

// 2. 使用 MethodHandle（Java 7+）
MethodHandles.Lookup lookup = MethodHandles.lookup();
MethodHandle handle = lookup.findVirtual(User.class, "getName", MethodType.methodType(String.class));
String name = (String) handle.invoke(user);

// 3. 使用字节码生成（如 ASM、ByteBuddy）完全避免反射
```

## 三、动态代理

### 3.1 代理模式

代理模式为对象提供代理，控制对原对象的访问。

```
客户端 → 代理对象 → 目标对象
```

**静态代理**：代理类在编译时确定，每个目标类需要单独的代理类。

**动态代理**：代理类在运行时动态生成，一个代理类可以代理多个目标类。

### 3.2 JDK 动态代理

JDK 动态代理基于**接口**，使用 `java.lang.reflect.Proxy` 实现。

```java
public interface UserService {
    void addUser(String name);
    void deleteUser(String name);
}

public class UserServiceImpl implements UserService {
    @Override
    public void addUser(String name) {
        System.out.println("添加用户: " + name);
    }

    @Override
    public void deleteUser(String name) {
        System.out.println("删除用户: " + name);
    }
}

// 创建代理
public class LogInvocationHandler implements InvocationHandler {
    private Object target;  // 目标对象

    public LogInvocationHandler(Object target) {
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("调用方法: " + method.getName());
        Object result = method.invoke(target, args);  // 调用目标方法
        System.out.println("方法调用完成");
        return result;
    }
}

// 使用
UserService realService = new UserServiceImpl();
UserService proxy = (UserService) Proxy.newProxyInstance(
    realService.getClass().getClassLoader(),
    realService.getClass().getInterfaces(),
    new LogInvocationHandler(realService)
);

proxy.addUser("张三");
// 输出：
// 调用方法: addUser
// 添加用户: 张三
// 方法调用完成
```

### 3.3 CGLIB 动态代理

CGLIB 基于**继承**，通过字节码生成目标类的子类。可以代理没有接口的类。

```java
public class UserService {
    public void addUser(String name) {
        System.out.println("添加用户: " + name);
    }
}

// CGLIB 代理
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(UserService.class);
enhancer.setCallback(new MethodInterceptor() {
    @Override
    public Object intercept(Object obj, Method method, Object[] args, MethodProxy proxy) throws Throwable {
        System.out.println("调用方法: " + method.getName());
        return proxy.invokeSuper(obj, args);  // 调用父类方法
    }
});

UserService proxy = (UserService) enhancer.create();
proxy.addUser("李四");
```

### 3.4 JDK 动态代理 vs CGLIB

| 对比项 | JDK 动态代理 | CGLIB |
|--------|-------------|-------|
| 代理方式 | 基于接口 | 基于继承 |
| 代理类 | 实现接口 | 继承目标类 |
| 限制 | 目标类必须实现接口 | 不能代理 final 类和方法 |
| 性能 | 生成代理快，调用慢 | 生成代理慢，调用快（有缓存后差距不大） |
| Spring 默认 | JDK 8 后默认 CGLIB | 优先使用 |

## 四、注解机制

### 4.1 注解的本质

注解是**一种特殊的接口**，继承自 `java.lang.annotation.Annotation`。

```java
// 定义注解
@Target(ElementType.METHOD)        // 注解可用在哪里
@Retention(RetentionPolicy.RUNTIME) // 注解保留到什么时候
public @interface MyAnnotation {
    String value() default "";
    int priority() default 0;
}

// 使用注解
@MyAnnotation(value = "test", priority = 1)
public void doSomething() { }
```

### 4.2 元注解

元注解是注解的注解，用于定义注解的行为：

| 元注解 | 说明 | 取值 |
|--------|------|------|
| `@Target` | 注解可用在哪里 | TYPE, FIELD, METHOD, PARAMETER, CONSTRUCTOR, etc. |
| `@Retention` | 注解保留策略 | SOURCE, CLASS, RUNTIME |
| `@Documented` | 注解会被 Javadoc 记录 | - |
| `@Inherited` | 注解可被子类继承 | - |
| `@Repeatable` | 注解可重复使用（Java 8+） | - |

**@Retention 三种策略**：

| 策略 | 保留范围 | 使用场景 |
|------|---------|---------|
| SOURCE | 仅源码，编译后丢弃 | `@Override`, `@SuppressWarnings`（编译检查） |
| CLASS | 编译后保留在 class 文件，但 JVM 忽略 | 字节码处理工具 |
| RUNTIME | 运行时保留，可通过反射读取 | Spring、Retrofit 等框架 |

### 4.3 运行时解析注解

```java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface Table {
    String name();
}

@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Column {
    String name();
    boolean nullable() default true;
}

// 使用注解
@Table(name = "t_user")
public class User {
    @Column(name = "id", nullable = false)
    private Long id;

    @Column(name = "username")
    private String name;
}

// 解析注解
public class AnnotationParser {
    public static String getTableName(Class<?> clazz) {
        Table table = clazz.getAnnotation(Table.class);
        return table != null ? table.name() : clazz.getSimpleName().toLowerCase();
    }

    public static List<String> getColumnNames(Class<?> clazz) {
        List<String> columns = new ArrayList<>();
        for (Field field : clazz.getDeclaredFields()) {
            Column column = field.getAnnotation(Column.class);
            if (column != null) {
                columns.add(column.name());
            }
        }
        return columns;
    }
}
```

## 五、APT（编译时注解处理）

### 5.1 为什么用 APT

运行时反射的问题：
- **性能开销**：每次都需要反射调用
- **编译期无检查**：错误只能在运行时发现

APT（Annotation Processing Tool）在编译时处理注解，生成代码，避免运行时反射。

### 5.2 APT 工作原理

```
源代码（带注解）
    ↓
编译器调用注解处理器
    ↓
处理器分析注解，生成 Java 代码
    ↓
生成的代码参与编译
    ↓
最终 class 文件
```

### 5.3 APT 示例：BindView

```java
// 1. 定义注解
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.SOURCE)  // 只在源码保留
public @interface BindView {
    int value();  // view id
}

// 2. 定义注解处理器
@SupportedAnnotationTypes("com.example.BindView")
@SupportedSourceVersion(SourceVersion.RELEASE_8)
public class BindViewProcessor extends AbstractProcessor {

    @Override
    public boolean process(Set<? extends TypeElement> annotations, RoundEnvironment roundEnv) {
        // 按 Element 分组
        Map<Element, List<VariableElement>> map = new HashMap<>();

        for (Element element : roundEnv.getElementsAnnotatedWith(BindView.class)) {
            VariableElement field = (VariableElement) element;
            Element enclosingClass = field.getEnclosingElement();
            map.computeIfAbsent(enclosingClass, k -> new ArrayList<>()).add(field);
        }

        // 为每个类生成绑定代码
        for (Map.Entry<Element, List<VariableElement>> entry : map.entrySet()) {
            generateBindingClass(entry.getKey(), entry.getValue());
        }

        return true;
    }

    private void generateBindingClass(Element classElement, List<VariableElement> fields) {
        String className = classElement.getSimpleName().toString();
        String packageName = processingEnv.getElementUtils().getPackageOf(classElement).toString();
        String binderClassName = className + "_ViewBinding";

        // 生成 Java 代码
        JavaFile javaFile = JavaFile.builder(packageName,
            TypeSpec.classBuilder(binderClassName)
                .addMethod(MethodSpec.constructorBuilder()
                    .addParameter(ClassName.get(packageName, className), "target")
                    .addStatement("$T view = target.findViewById", ClassName.get("android.view", "View"))
                    .addStatement("target.name = ($T) view.findViewById($L)",
                        ClassName.get("android.widget", "TextView"), 123)
                    .build())
                .build())
            .build();

        try {
            javaFile.writeTo(processingEnv.getFiler());
        } catch (IOException e) {
            processingEnv.getMessager().printMessage(Diagnostic.Kind.ERROR, e.getMessage());
        }
    }
}

// 3. 注册处理器（resources/META-INF/services/javax.annotation.processing.Processor）
// com.example.BindViewProcessor

// 4. 使用
public class MainActivity extends AppCompatActivity {
    @BindView(R.id.tv_name)
    TextView nameView;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);
        MainActivity_ViewBinding.bind(this);  // 生成的绑定类
    }
}
```

| 框架 | 用途 | 原理 |
|------|------|------|
| ButterKnife | View 绑定 | APT + 运行时反射 |
| Dagger2 | 依赖注入 | 纯 APT |
| Room | 数据库 | APT |
| Glide | 图片加载 | APT |
| EventBus | 事件总线 | APT |

## 六、KSP（Kotlin Symbol Processing）

### 6.1 KSP vs KAPT

| 对比项 | KAPT | KSP |
|--------|------|-----|
| 原理 | 生成 Java 存根（Stub） | 直接分析 Kotlin 语法树 |
| 速度 | 慢（需生成存根） | 快 2x+ |
| 支持 | Java 注解处理器兼容 | 原生 Kotlin 支持 |
| 生成代码 | Java | Java/Kotlin |

### 6.2 迁移建议

KSP 是 KAPT 的替代方案，Google 推荐 Kotlin 项目使用 KSP：

- Room、Moshi、Hilt 等已支持 KSP
- 编译速度显著提升
- 未来 KAPT 将被弃用

## 七、常见面试题与解答

### Q1：反射的应用场景有哪些？

**答**：

1. **框架开发**：Spring IOC（依赖注入）、AOP（动态代理）
2. **序列化框架**：Gson、Jackson 解析 JSON
3. **ORM 框架**：Hibernate、MyBatis 映射数据库
4. **测试框架**：JUnit 调用测试方法
5. **热修复/插件化**：动态加载类、修改字段

---

### Q2：JDK 动态代理和 CGLIB 有什么区别？

**答**：

| 对比项 | JDK 动态代理 | CGLIB |
|--------|-------------|-------|
| 实现方式 | 基于接口，代理类实现接口 | 基于继承，代理类继承目标类 |
| 限制 | 必须实现接口 | 不能代理 final 类/方法 |
| 原理 | 反射调用 | 字节码生成子类 |
| 性能 | 生成快，调用慢 | 生成慢，调用快 |

Spring Boot 2.x 默认使用 CGLIB。

---

### Q3：注解的 @Retention 有哪些策略？如何选择？

**答**：

| 策略 | 保留到 | 使用场景 |
|------|--------|---------|
| SOURCE | 源码 | 编译检查（@Override, @SuppressWarnings） |
| CLASS | 字节码 | 字节码处理工具 |
| RUNTIME | 运行时 | 框架运行时读取（Spring, Retrofit） |

选择：
- 仅编译时需要 → SOURCE
- 运行时需要反射读取 → RUNTIME
- 字节码处理工具使用 → CLASS

---

### Q4：APT 相比运行时反射有什么优势？

**答**：

1. **性能更好**：生成的代码直接调用，无反射开销
2. **编译期检查**：错误在编译时发现，而非运行时崩溃
3. **类型安全**：生成的代码是强类型的
4. **可读性**：可以查看生成的代码，便于调试

缺点是增加编译时间和代码量。

---

### Q5：如何避免反射的性能问题？

**答**：

1. **缓存反射对象**：避免重复查找 Method/Field
2. **使用 MethodHandle**：Java 7+ 提供的轻量级反射替代
3. **使用 APT**：编译时生成代码，避免运行时反射
4. **字节码生成**：使用 ASM、ByteBuddy 直接生成字节码

---

### Q6：什么是 Java Bean 内省（Introspection）？

**答**：内省是 Java 对反射的封装，专门用于操作 JavaBean。通过 `Introspector` 获取 `BeanInfo`，再获取属性描述符 `PropertyDescriptor`。

```java
BeanInfo beanInfo = Introspector.getBeanInfo(User.class);
for (PropertyDescriptor pd : beanInfo.getPropertyDescriptors()) {
    System.out.println(pd.getName());  // 属性名
    Method readMethod = pd.getReadMethod();  // getter
    Method writeMethod = pd.getWriteMethod(); // setter
}
```

内省比反射更安全、更规范，适合操作符合 JavaBean 规范的对象。

---

### Q7：KSP 为什么比 KAPT 快？

**答**：

- **KAPT**：需要先为 Kotlin 代码生成 Java 存根（Stub），然后 Java 注解处理器处理存根，整个过程需要多次编译
- **KSP**：直接分析 Kotlin 的语法树（PSI），不需要生成存根，一遍完成

KSP 减少了中间步骤，因此编译速度更快。
