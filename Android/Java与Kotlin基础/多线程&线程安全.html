Android 并发编程与线程安全 Mastery 手册

版本： 1.0.0 | 适用对象： Android 中高级开发工程师 | 目标： 构建完整的并发知识体系，深入理解底层原理

📘 手册说明： 本手册不仅涵盖 API 使用，更侧重于 Why (设计原因) 和 How (底层实现)。建议结合 Android 源码 (Framework) 阅读以达到最佳效果。

1. 前言与设计哲学

Android 的并发模型建立在 Java 并发基础之上，但针对移动设备的特性（UI 渲染、电池、生命周期）做了大量定制。理解其设计哲学是避免并发错误的第一步。

1.1 为什么 UI 是单线程的？

Android UI 工具包（View 体系）不是线程安全的。如果允许多线程操作 UI，将引入复杂的锁机制，导致：

  • 死锁风险增加：多个线程互相等待锁。
  • 性能开销大：锁竞争导致上下文切换频繁。
  • 调试困难：竞态条件难以复现。

设计决策：采用 单线程渲染模型。所有 UI 操作必须在主线程（Main Thread），耗时操作必须在子线程（Worker Thread）。

1.2 线程切换的代价

线程不是越多越好。线程切换涉及：

  1. 保存当前线程上下文（寄存器、程序计数器）。
  2. 加载新线程上下文。
  3. 刷新 CPU 缓存（Cache Miss）。

频繁切换会导致 CPU 时间片浪费在调度上，而非业务逻辑。

2. 并发理论基础 (JMM)

Java 内存模型 (JMM) 定义了线程如何与主内存和工作内存交互。并发问题的根源在于：原子性、可见性、有序性 无法得到保证。

2.1 三大特性详解

特性 定义 问题示例 解决方案
原子性 操作不可中断，要么全做要么全不做。 i++ (读 - 改 - 写) synchronized, Lock, Atomic
可见性 线程修改变量，其他线程立即可见。 死循环读取不到新值 volatile, synchronized, Lock
有序性 程序执行顺序与代码顺序一致。 指令重排序导致初始化未完成 volatile (禁止重排), Happens-Before

2.2 Happens-Before 规则

JMM 定义了一系列规则来判断操作之间的可见性。核心规则包括：

  • 程序顺序规则：一个线程内，按照代码顺序，书写在前面的操作先行发生于书写在后面的操作。
  • volatile 变量规则：对一个 volatile 变量的写操作先行发生于后面对这个变量的读操作。
  • 传递性：如果 A happens-before B，且 B happens-before C，那么 A happens-before C。

3. Android 消息机制 (Handler)

Handler 是 Android 线程间通信的核心。理解它对于理解 UI 刷新、AsyncTask 底层、协程调度器至关重要。

3.1 核心组件职责

  • Thread：线程载体，每个线程拥有独立的 Looper。
  • Looper：消息循环器，内部维护死循环 loop()，不断从 MessageQueue 取消息。
  • MessageQueue：消息队列，底层采用单链表结构，支持同步屏障（Sync Barrier）优先处理异步消息。
  • Handler：消息发送者 (sendMessage) 和处理者 (handleMessage)。

3.2 消息流转源码分析

Looper.loop() 核心逻辑：

// frameworks/base/core/java/android/os/Looper.java
public static void loop() {
    final Looper me = myLooper();
    if (me == null) throw new RuntimeException("No Looper...");
    final MessageQueue queue = me.mQueue;

    for (;;) {
        // 1. 获取消息。若无消息，nativePollOnce 会阻塞线程，不消耗 CPU
        Message msg = queue.next(); 
        if (msg == null) return; // 退出循环

        // 2. 标记消息开始处理时间 (用于监控 ANR)
        msg.target.dispatchMessage(msg);

        // 3. 回收消息
        msg.recycleUnchecked();
    }
}

设计原因：为什么需要死循环？因为线程执行完 run() 默认会销毁。UI 线程需要永久存活以响应用户交互，loop() 保证线程不退出，且在无消息时通过 native 层阻塞（epoll 机制）节省电量。

3.3 同步屏障 (Sync Barrier)

Android 为了保证 UI 流畅，引入了同步屏障。当屏障存在时，MessageQueue 会跳过所有同步消息，优先处理异步消息（如 VSync 信号驱动的 Choreographer）。

  • 应用场景：UI 渲染优先级高于普通消息。
  • API：MessageQueue.postSyncBarrier() (隐藏 API，框架内部使用)。

4. 锁机制与底层原理

锁是保证线程安全最直接的手段。Android 中主要使用 synchronized 和 ReentrantLock。

4.1 synchronized 锁升级过程

JVM 为了减少锁开销，实现了偏向锁 -> 轻量级锁 -> 重量级锁的升级过程。

flowchart LR A[无锁] -->|第一个线程访问 | B(偏向锁) B -->|另一个线程竞争 | C(轻量级锁) C -->|自旋失败/竞争激烈 | D(重量级锁) D -->|OS Mutex| E[线程阻塞] style A fill:#e8f5e9,stroke:#2e7d32 style B fill:#fff3e0,stroke:#ef6c00 style C fill:#e3f2fd,stroke:#1565c0 style D fill:#ffebee,stroke:#c62828
  • 偏向锁：记录线程 ID。若无竞争，无需 CAS 操作。
  • 轻量级锁：使用 CAS 自旋。若短时间获得锁，避免阻塞。
  • 重量级锁：依赖操作系统互斥量 (Mutex)，线程挂起，开销大。

4.2 AQS (AbstractQueuedSynchronizer)

ReentrantLock, CountDownLatch, Semaphore 的基石。

  • 核心状态：volatile int state。0 表示无锁，>0 表示有锁（可重入次数）。
  • 队列：FIFO 双向队列 (CLH 变体)，存储等待线程。
  • 获取锁：CAS 尝试修改 state。失败则封装成 Node 入队，调用 LockSupport.park() 阻塞。
  • 释放锁：修改 state，调用 LockSupport.unpark() 唤醒后继节点。

5. 原子类与 CAS

基于硬件指令的无锁并发方案，性能高于锁，但高竞争下消耗 CPU。

5.1 CAS (Compare-And-Swap)

包含三个操作数：内存位置 V、预期原值 A、新值 B。仅当 V==A 时，将 V 更新为 B。

ABA 问题： 值从 A 变 B 又变回 A，CAS 误判成功。
解决： 使用 AtomicStampedReference 增加版本号。

5.2 LongAdder (高性能计数器)

在 Android 高并发计数场景（如统计埋点），AtomicLong 可能成为瓶颈。LongAdder 采用分段累加，最后汇总，减少热点竞争。

6. 线程隔离 (ThreadLocal)

让每个线程拥有独立的变量副本，避免共享竞争。

6.1 底层结构

每个 Thread 对象内部维护一个 ThreadLocalMap。

  • Key：ThreadLocal 对象本身（弱引用）。
  • Value：存储的数据（强引用）。

6.2 内存泄漏风险

classDiagram class Thread { ThreadLocalMap threadLocals } class ThreadLocalMap { Entry[] table } class Entry { WeakReference~ThreadLocal~ key Object value } Thread --> ThreadLocalMap ThreadLocalMap ..> Entry note for Entry "Key 为 null 时\nValue 仍被强引用\n若线程存活则泄漏"

泄漏原因：若线程长期存活（如线程池），Key 被 GC 回收变为 null，但 Value 仍被 Map 强引用，无法回收。

最佳实践：使用完毕后必须调用 threadLocal.remove()。

Android 应用：Looper.sThreadLocal 确保每个线程只有一个 Looper。

7. 线程池最佳实践

避免手动创建 new Thread()，使用线程池复用资源。

7.1 ThreadPoolExecutor 核心参数

参数 含义 调优建议
corePoolSize 核心线程数 CPU 密集型：CPU 核数 +1；IO 密集型：2 * CPU 核数
maximumPoolSize 最大线程数 防止资源耗尽，需结合系统内存评估
workQueue 任务队列 建议使用有界队列 (如 ArrayBlockingQueue)，防止 OOM
RejectedExecutionHandler 拒绝策略 默认 AbortPolicy 抛异常。可自定义日志或丢弃策略

7.2 Android 中的线程池

  • AsyncTask：内部使用静态线程池，串行/并行策略随版本变化，已废弃。
  • Glide：使用固定的核心线程池处理图片加载。
  • 推荐：根据业务场景自定义线程池，或使用协程调度器。

8. Kotlin 协程 (现代方案)

协程是轻量级的线程，基于挂起函数实现非阻塞异步。

8.1 结构化并发 (Structured Concurrency)

协程必须在一个 CoroutineScope 中启动。Scope 销毁时，所有子协程自动取消。

优势： 彻底解决“页面销毁了，后台线程还在跑”导致的崩溃和泄漏。

8.2 常用调度器 (Dispatchers)

  • Dispatchers.Main：主线程，UI 操作。
  • Dispatchers.IO：磁盘/网络 IO，默认 64 个线程。
  • Dispatchers.Default：CPU 密集型计算，默认等于 CPU 核心数。
  • Dispatchers.Unconfined：不限线程，仅用于特定场景。

8.3 代码示例

// ViewModel 中推荐写法
viewModelScope.launch {
    // 1. 主线程：更新 UI
    uiState.value = Loading
    
    try {
        // 2. 切换到 IO 线程：网络请求
        val data = withContext(Dispatchers.IO) {
            repository.fetchData()
        }
        
        // 3. 自动切回主线程：更新结果
        uiState.value = Success(data)
    } catch (e: Exception) {
        // 4. 异常处理
        uiState.value = Error(e.message)
    }
}
// viewModelScope 销毁时，上述所有任务自动取消

9. 常见陷阱与调试

9.1 ANR (Application Not Responding)

  • 定义：主线程阻塞超过 5 秒 (输入事件) 或 10 秒 (广播)。
  • 检测：开启开发者选项“显示 CPU 使用情况”，或使用 StrictMode。
  • 排查：分析 /data/anr/traces.txt，查找主线程堆栈。

9.2 内存泄漏自查表

  • [ ] Handler 是否为静态内部类？是否移除了消息？
  • [ ] ThreadLocal 是否调用了 remove()？
  • [ ] 单例是否持有了 Activity Context？(应使用 Application)
  • [ ] 协程是否使用了正确的 Scope (lifecycleScope/viewModelScope)？
  • [ ] 静态集合是否清除了 Activity 引用？

9.3 调试工具

  • Android Profiler：查看 CPU 线程活动，识别阻塞。
  • StrictMode：检测主线程网络/磁盘操作。
  • BlockCanary：第三方库，监控主线程卡顿。
