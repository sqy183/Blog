# Java & Android 类加载机制深度解析

## 一、概述

类加载（Class Loading）是 JVM 将 `.class` 字节码文件加载到内存，并转化为运行时 `java.lang.Class` 对象的过程。理解类加载机制是掌握热修复、插件化、模块化等 Android 高级技术的基础。

**核心要点：**
- Java 采用**按需加载** — 类在首次被使用时才会触发加载，而非启动时全部加载
- 类加载由 `ClassLoader`（类加载器）负责，不同的类加载器负责不同来源的类
- **双亲委派模型**是 ClassLoader 协同工作的核心规则
- Android 基于 JVM 类加载思想，但用 Dalvik/ART 替代了标准 JVM，有自己的 ClassLoader 体系

## 二、类的生命周期

一个类从加载到卸载，完整生命周期如下：

```
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
       └──── 连接（Linking） ────┘
```

| 阶段 | 说明 |
|------|------|
| **加载（Loading）** | 通过类的全限定名找到 .class 字节流，将其转化为方法区的运行时数据结构，在堆中生成 `Class` 对象 |
| **验证（Verification）** | 校验字节码的合法性（文件格式、元数据、字节码指令、符号引用），防止恶意代码 |
| **准备（Preparation）** | 为类变量（static 字段）分配内存并设置**零值**（不是代码中的初始值） |
| **解析（Resolution）** | 将常量池中的符号引用替换为直接引用（内存地址） |
| **初始化（Initialization）** | 执行类构造器 `<clinit>()`，即 static 代码块和 static 变量赋值。**这是类加载过程中唯一执行用户代码的阶段** |

> **准备阶段 vs 初始化阶段的经典区别**：
> ```java
> public static int value = 123;
> // 准备阶段：value = 0（零值）
> // 初始化阶段：value = 123（执行赋值）
>
> public static final int CONST = 123;
> // 准备阶段：CONST = 123（final + 编译期常量，直接赋值）
> ```

### 触发类初始化的时机（有且仅有以下 5 种）

1. `new` 对象、读写 static 字段（非 final 编译期常量）、调用 static 方法
2. 反射调用（`Class.forName()`）
3. 初始化子类时，父类尚未初始化
4. JVM 启动时的主类（包含 `main()` 的类）
5. JDK 7+ 的 MethodHandle 解析结果为特定句柄时

> **不会触发初始化的典型场景**：
> - 通过子类引用父类的 static 字段（只初始化父类）
> - 创建类的数组（`new MyClass[10]`）
> - 引用类的 `static final` 编译期常量

## 三、JVM 类加载器体系

### 3.1 三层类加载器

```
┌─────────────────────────────────┐
│   Bootstrap ClassLoader         │  C++ 实现，不继承 ClassLoader
│   加载 $JAVA_HOME/lib 核心类     │  如 rt.jar（java.lang.*、java.util.* 等）
└──────────────┬──────────────────┘
               │ 父加载器
┌──────────────▼──────────────────┐
│   Extension ClassLoader         │  Java 实现 (sun.misc.Launcher$ExtClassLoader)
│   加载 $JAVA_HOME/lib/ext 扩展类 │
└──────────────┬──────────────────┘
               │ 父加载器
┌──────────────▼──────────────────┐
│   Application ClassLoader       │  Java 实现 (sun.misc.Launcher$AppClassLoader)
│   加载 classpath 下的应用类       │  即我们写的业务代码
└─────────────────────────────────┘
```

> **注意**：这里的"父子关系"不是类继承关系，而是**组合关系**（每个 ClassLoader 持有一个 `parent` 字段指向上级加载器）。

### 3.2 ClassLoader 核心 API

```java
public abstract class ClassLoader {
    private final ClassLoader parent;  // 父加载器（组合关系）

    // 加载类的入口（实现了双亲委派逻辑）
    public Class<?> loadClass(String name) throws ClassNotFoundException;

    // 子类重写此方法来定义自己的加载逻辑
    protected Class<?> findClass(String name) throws ClassNotFoundException;

    // 将字节数组转化为 Class 对象（由 native 实现，通常在 findClass 中调用）
    protected final Class<?> defineClass(String name, byte[] b, int off, int len);
}
```

## 四、双亲委派模型（Parent Delegation Model）

### 4.1 核心机制

双亲委派是 `ClassLoader.loadClass()` 中实现的类加载规则：

```
加载请求："加载 com.example.MyClass"
        │
        ▼
 Application ClassLoader
   "我先不加载，问问我爸"
        │
        ▼
  Extension ClassLoader
   "我也先不加载，问问我爸"
        │
        ▼
  Bootstrap ClassLoader
   "我的范围里找不到这个类"
        │ 加载失败，返回
        ▼
  Extension ClassLoader
   "我的范围里也找不到"
        │ 加载失败，返回
        ▼
 Application ClassLoader
   "轮到我了，在 classpath 找到了，加载！"
        │
        ▼
     加载成功 ✓
```

**一句话概括：先委派父加载器尝试加载，父加载器加载不了时才自己加载。**

### 4.2 源码实现

```java
// ClassLoader.java（简化）
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    // 1. 检查是否已经加载过
    Class<?> c = findLoadedClass(name);

    if (c == null) {
        try {
            // 2. 委派给父加载器
            if (parent != null) {
                c = parent.loadClass(name, false);
            } else {
                // parent 为 null 说明父加载器是 Bootstrap
                c = findBootstrapClassOrNull(name);
            }
        } catch (ClassNotFoundException e) {
            // 父加载器无法加载，不抛异常，继续往下走
        }

        if (c == null) {
            // 3. 父加载器加载失败，自己尝试加载
            c = findClass(name);  // 子类重写此方法
        }
    }

    return c;
}
```

### 4.3 为什么需要双亲委派？

| 目的 | 说明 |
|------|------|
| **安全性** | 防止用户自定义的类替换核心类。即使你写了一个 `java.lang.String`，也不会被加载，因为 Bootstrap ClassLoader 会优先加载 JDK 自带的 String |
| **唯一性** | 保证同一个类在 JVM 中只会被加载一次。判断两个类是否"相同"，不仅看类名，还要看是否由**同一个 ClassLoader** 加载 |
| **层次清晰** | 每层加载器各司其职，核心库、扩展库、应用代码互不干扰 |

> **关键认知**：JVM 中判断两个类是否相同的条件是 **全限定名 + 加载它的 ClassLoader 实例** 都相同。即使字节码完全一样，不同 ClassLoader 加载出来的是两个不同的类，相互之间不能强制类型转换。

## 五、打破双亲委派

双亲委派不是强制规范，而是推荐约定。有些场景下必须打破它。

### 5.1 SPI 机制（Service Provider Interface）

**问题**：JDBC 的 `DriverManager` 在 `rt.jar` 中，由 Bootstrap ClassLoader 加载。但具体的数据库驱动（如 `com.mysql.jdbc.Driver`）在应用 classpath 下，Bootstrap 加载器找不到。

**解决**：使用**线程上下文类加载器（Thread Context ClassLoader）**。

```java
// DriverManager 中的关键逻辑（简化）
// 虽然 DriverManager 自己是 Bootstrap 加载的
// 但它通过线程上下文拿到 Application ClassLoader 去加载驱动
ClassLoader cl = Thread.currentThread().getContextClassLoader();
Class.forName("com.mysql.jdbc.Driver", true, cl);
```

```
正常双亲委派：子 → 父（向上委派）
SPI 场景：    父 → 子（向下委托）← 打破了双亲委派
```

### 5.2 Tomcat 的类加载

Tomcat 需要在同一个 JVM 中部署多个 Web 应用，每个应用可能依赖不同版本的同一个库。

```
      Bootstrap ClassLoader
              │
       Extension ClassLoader
              │
      Application ClassLoader
              │
       Tomcat Common ClassLoader       ← Tomcat 共享类
          ┌───┴───────┐
   WebApp1 ClassLoader  WebApp2 ClassLoader   ← 各自隔离
   (lib v1.0)           (lib v2.0)
```

每个 WebApp ClassLoader **优先自己加载**（先 findClass，再委派父加载器），打破了双亲委派的"先问爸"原则，实现了不同应用间的类隔离。

### 5.3 热修复 / 插件化（Android 场景）

Android 的热修复框架（如 Tinker、Sophix）和插件化框架（如 RePlugin、Shadow）都需要打破或绕过双亲委派：

- **热修复**：将修复后的 dex 插入到 `DexPathList.dexElements` 数组的**最前面**，利用 ClassLoader 的"找到即返回"特性，让修复类优先于 bug 类被加载
- **插件化**：为每个插件创建独立的 `DexClassLoader`，加载插件 APK 中的类，实现插件与宿主的类隔离

## 六、Android 的类加载器体系

Android 使用 Dalvik/ART 虚拟机，加载的是 `.dex` 文件而非 `.class` 文件，因此有自己的 ClassLoader 实现。

### 6.1 类加载器层次

```
BootClassLoader                        ← Java 实现（非 C++），加载 Framework 核心类
       │
BaseDexClassLoader                     ← 抽象基类，核心加载逻辑在此
    ├── PathClassLoader                ← 加载已安装 APK 的类（系统默认使用）
    ├── DexClassLoader                 ← 可加载任意路径的 dex/apk/jar
    └── InMemoryDexClassLoader         ← Android 8.0+，从内存加载 dex
```

> **与 JVM 的关键区别**：Android 没有 Extension ClassLoader，`BootClassLoader` 是 Java 层实现（而非 JVM 的 C++ Bootstrap ClassLoader）。

### 6.2 BaseDexClassLoader 加载原理

```java
// BaseDexClassLoader.java（简化）
public class BaseDexClassLoader extends ClassLoader {
    private final DexPathList pathList;  // 核心！管理所有 dex 文件

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        Class c = pathList.findClass(name, suppressedExceptions);
        if (c == null) {
            throw new ClassNotFoundException(name);
        }
        return c;
    }
}
```

```java
// DexPathList.java（简化）
public Class<?> findClass(String name, List<Throwable> suppressed) {
    // 遍历 dexElements 数组，从前到后查找
    for (Element element : dexElements) {
        Class<?> clazz = element.findClass(name, definingContext, suppressed);
        if (clazz != null) {
            return clazz;  // 找到就返回，不再继续查找
        }
    }
    return null;
}
```

**关键数据结构 — dexElements 数组：**

```
DexPathList.dexElements[]
┌───────────┬───────────┬───────────┬───────────┐
│  dex 文件1 │  dex 文件2 │  dex 文件3 │    ...    │
└───────────┴───────────┴───────────┴───────────┘
     ↑                        ↑
  优先查找                  后查找

热修复原理：将 patch.dex 插入到数组最前面
┌───────────┬───────────┬───────────┬───────────┐
│ patch.dex │  dex 文件1 │  dex 文件2 │    ...    │
└───────────┴───────────┴───────────┴───────────┘
     ↑
  修复后的类优先被加载
```

### 6.3 PathClassLoader vs DexClassLoader

| 对比项 | PathClassLoader | DexClassLoader |
|--------|----------------|----------------|
| **用途** | 加载已安装的 APK | 加载外部的 dex/apk/jar |
| **optimizedDirectory** | 系统默认路径 | Android 8.0 前可自定义，8.0+ 废弃该参数 |
| **使用场景** | 系统为每个 App 自动创建 | 插件化、热修复框架手动创建 |
| **Android 8.0+ 差异** | 两者实际已无本质区别，都继承自 BaseDexClassLoader |

> **Android 8.0+ 的变化**：`DexClassLoader` 的 `optimizedDirectory` 参数被忽略，两者行为趋于一致。但概念上仍建议用 `PathClassLoader` 加载已安装的类，`DexClassLoader` 加载动态下发的类。

### 6.4 与 Zygote 预加载的联系

回到 App 启动流程中 Zygote 的预加载机制：

```
Zygote 进程启动
  → preloadClasses()                    // 使用 BootClassLoader
    → Class.forName("android.app.Activity")
    → Class.forName("android.view.View")
    → ... (4000+ 类)

Zygote fork App 进程
  → 子进程继承 BootClassLoader 及其已加载的所有类（COW共享）
  → 系统为 App 创建 PathClassLoader 加载 APK 中的业务类
```

```
App 进程中的 ClassLoader 链：

  PathClassLoader (parent=BootClassLoader)
       │
       │ loadClass("com.example.MyActivity")
       │   1. findLoadedClass → 没有
       │   2. parent.loadClass → BootClassLoader 找不到
       │   3. findClass → 在 dexElements 中找到，加载成功
       │
       │ loadClass("android.app.Activity")
       │   1. findLoadedClass → 没有（本 ClassLoader 没加载过）
       │   2. parent.loadClass → BootClassLoader 已加载过，直接返回
       │   （Zygote 预加载的类在这里被复用）
```

这就是双亲委派在 Android 启动过程中的实际体现：App 的业务类由 `PathClassLoader` 加载，Framework 类委派给 `BootClassLoader`（Zygote 已预加载）。

## 七、常见面试题与解答

### Q1：什么是双亲委派模型？为什么需要它？

**答**：双亲委派是 ClassLoader 的工作规则 — 收到类加载请求时，先委派给父加载器尝试，父加载器失败后才自己加载。

作用有三：
1. **安全性**：防止自定义类冒充核心类（如伪造 `java.lang.String`）
2. **唯一性**：同一个类只会被加载一次，避免出现"同名不同类"的问题
3. **层次分离**：核心类、扩展类、应用类由不同层级的加载器负责，职责清晰

---

### Q2：如何打破双亲委派？有哪些实际场景？

**答**：重写 `loadClass()` 方法即可改变委派逻辑（而非重写 `findClass()`）。

实际场景：
- **SPI 机制**（JDBC、SLF4J）：核心接口在 Bootstrap 层，实现类在应用层，通过线程上下文 ClassLoader 实现"父加载器委托子加载器"
- **Tomcat**：每个 WebApp 有独立的 ClassLoader，优先自己加载以实现应用间类隔离
- **OSGi**：网状的类加载结构，模块间可以互相委托
- **Android 热修复**：将修复类所在的 dex 插入 dexElements 数组前端，优先于原始类被加载

---

### Q3：Android 的 ClassLoader 和 JVM 的有什么区别？

**答**：

| 对比项 | JVM | Android (ART/Dalvik) |
|--------|-----|---------------------|
| **加载文件** | `.class` 字节码 | `.dex` 文件 |
| **顶层加载器** | Bootstrap ClassLoader（C++实现，不可见） | BootClassLoader（Java实现，可访问） |
| **应用加载器** | AppClassLoader | PathClassLoader |
| **核心数据结构** | classpath 查找 | `DexPathList.dexElements[]` 数组遍历 |
| **中间层** | ExtClassLoader | 无 |

但双亲委派的核心思想是一致的。

---

### Q4：如何理解"同一个类被不同 ClassLoader 加载后是不同的类"？

**答**：JVM 判断类是否相同的标准是 **全限定名 + ClassLoader 实例** 的组合。

```java
ClassLoader cl1 = new MyClassLoader("path/to/classes");
ClassLoader cl2 = new MyClassLoader("path/to/classes");

Class<?> c1 = cl1.loadClass("com.example.Foo");
Class<?> c2 = cl2.loadClass("com.example.Foo");

c1 == c2          // false！不同 ClassLoader 加载的是不同的 Class 对象
c1.equals(c2)     // false
c1 instanceof Foo  // 取决于 Foo 是由哪个 ClassLoader 定义的
```

这个特性是 Tomcat 多应用隔离、Android 插件化的基础 — 不同 ClassLoader 加载的同名类互不干扰。但也会导致跨 ClassLoader 传递对象时出现 `ClassCastException`。

---

### Q5：Android 热修复的类加载方案原理是什么？

**答**：以 Tinker 为代表的类加载方案，核心步骤：

1. 服务端下发差分包（patch.dex）
2. 客户端合成完整的修复 dex 文件
3. 通过反射拿到 `PathClassLoader` → `BaseDexClassLoader.pathList` → `DexPathList.dexElements`
4. 将 patch.dex 对应的 Element 插入到 `dexElements` 数组**最前面**
5. 下次加载某个类时，ClassLoader 遍历 dexElements，在 patch.dex 中先找到修复后的类，直接返回，原始的 bug 类就不会被加载了

**限制**：已经加载到内存的类无法替换，因此这种方案通常需要**重启 App** 才能生效。

---

### Q6：类的加载过程中，准备阶段和初始化阶段有什么区别？

**答**：

- **准备阶段**：为 static 变量分配内存并赋**零值**（int→0, boolean→false, 引用→null）。`static final` 的编译期常量会直接赋真实值。
- **初始化阶段**：执行 `<clinit>()` 方法，即 static 代码块和 static 变量的赋值语句，按代码中出现的顺序执行。

```java
static int a = 10;
static final int B = 20;
static int c;
static { c = 30; }

// 准备阶段后：a=0, B=20, c=0
// 初始化阶段后：a=10, B=20, c=30
```

---

### Q7：Class.forName() 和 ClassLoader.loadClass() 有什么区别？

**答**：

| 方法 | 是否触发初始化 | 使用的 ClassLoader |
|------|-------------|-------------------|
| `Class.forName("X")` | **是**，默认执行 `<clinit>()` | 调用者的 ClassLoader |
| `Class.forName("X", false, cl)` | 第二个参数控制，false 则不初始化 | 指定的 ClassLoader |
| `cl.loadClass("X")` | **否**，只执行到加载阶段 | 指定的 ClassLoader |

典型区别场景：JDBC 注册驱动依赖 `Class.forName()` 触发 Driver 类的 static 代码块执行 `DriverManager.registerDriver()`，如果用 `ClassLoader.loadClass()` 则驱动不会注册。
