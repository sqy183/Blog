📱 Android 页面绘制全景解析

AOSP 源码级 - 类关系、事件流转与核心机制

文档版本：基于 AOSP Android 13 (android-13.0.0_r81)

核心主题：Activity → Window → DecorView → ViewRootImpl → Surface → BufferQueue → SurfaceFlinger 完整链路

关键机制：脏区域检测、VSync 调度、Buffer Holding、Triple Buffering、静态界面持续显示原理

🎯 核心结论速览

1. 静态界面持续显示原理

Buffer Holding 机制：SurfaceFlinger 持有已合成的 buffer，每帧重复使用同一帧数据。无新帧 ≠ 无显示，绘制（生产）与显示（消费）完全解耦。

2. VSync 按需分发

脏区域驱动：仅当窗口有脏区域（用户交互/动画/属性变更）时，EventThread 才分发 VSync 信号。静态界面无脏区域 → 不分发 → 应用零绘制（省电 90%+）。

3. BufferQueue 跨进程共享

单一 BufferQueueCore：Producer（App）与 Consumer（SurfaceFlinger）共享同一个 BufferQueueCore 实例，通过 Binder 传递指针实现零拷贝。64 个 slot 是元数据容器，实际 buffer 数量动态分配（2-3 个用于 UI，6-8 个用于 Camera）。

1. VSync 分发决策（脏区域检测）

// frameworks/native/services/surfaceflinger/SurfaceFlinger.cpp
Region SurfaceFlinger::computeDirtyRegion(const DisplayDevice& display) {
    Region dirty;
    for (const auto& layer : mDrawingState.layersSortedByZ) {
        if (layer->isVisible()) {
            dirty |= layer->getDirtyRegion(); // 合并所有 Layer 脏区域
        }
    }
    return dirty & display.getBounds();
}

// frameworks/native/services/surfaceflinger/Scheduler/EventThread.cpp
void EventThread::onHotplugReceived(...) {
    Region dirtyRegion = mFlinger->computeDirtyRegion(display);
    if (!dirtyRegion.isEmpty()) {
        dispatchVsyncToApps(dirtyRegion); // 有脏区域 → 分发 VSync
    } else {
        mFlinger->onMessageInvalidate(); // 无脏区域 → 仅 SF 自身合成
    }
}
✅ 静态界面省电原理： 无脏区域 → 不分发 VSync → App 零绘制（CPU/GPU 休眠）→ 仅 SF 合成旧帧（~1-2mW）。

2. Buffer 状态流转（BufferQueueCore.mSlots[]）

// frameworks/native/libs/gui/BufferQueueCore.h
namespace android::BufferQueueDefs {
    static constexpr int NUM_BUFFER_SLOTS = 64; // 固定 64 个 slot
}

enum BufferState { FREE, DEQUEUED, QUEUED, ACQUIRED };

struct BufferSlot {
    sp<GraphicBuffer> mGraphicBuffer; // 指向实际 GraphicBuffer（可能为 null）
    BufferState mBufferState;
    sp<Fence> mFence;
};

// 典型状态流转:
// dequeueBuffer(): FREE → DEQUEUED
// queueBuffer(): DEQUEUED → QUEUED → 触发 onFrameAvailable()
// acquireBuffer(): QUEUED → ACQUIRED (等待 acquireFence)
// releaseBuffer(): ACQUIRED → FREE (静态界面跳过此步)

3. 静态界面持续显示机制（Buffer Holding）

// frameworks/native/services/surfaceflinger/Layer.cpp
sp<GraphicBuffer> Layer::getBuffer() const {
    return mActiveBuffer; // ← 始终返回当前持有的 buffer
}

void Layer::onFrameAvailable(const BufferItem& item) {
    mActiveBuffer = item.mGraphicBuffer; // 新帧到来：更新 active buffer
    mDirtyRegion = mVisibleRegion;
}

void Layer::releasePendingBuffers() {
    // 仅当 mActiveBuffer 变更时才 release
    if (mActiveBuffer != mPreviousBuffer) {
        mConsumer->releaseBuffer(mPreviousBufferSlot, ...);
        mPreviousBuffer = mActiveBuffer;
    }
    // 无新帧时：mActiveBuffer 不变 → 不触发 releaseBuffer() → buffer 保持 ACQUIRED
}
❌ 常见误解： "无绘制 = 无显示" → 错误！ 绘制（生产）与显示（消费）解耦：无新帧时，消费者（SurfaceFlinger）重复使用旧帧，屏幕持续显示。

4. Fence 同步时序（硬件级同步）

Fence 类型 产生方 等待方 作用 源码位置
acquireFence GPU（渲染完成） SurfaceFlinger 确保 GPU 渲染完成后再读取 buffer RenderThread::queueBuffer()
releaseFence Display Controller（扫描完成） App（RenderThread） 确保 Display 读取完成后再重用 buffer HWComposer::presentDisplay()
// frameworks/native/libs/gui/BufferQueueProducer.cpp
status_t BufferQueueProducer::queueBuffer(int slot, ...) {
    sp<Fence> acquireFence = args.acquireFence;
    mSlots[slot].mAcquireFence = acquireFence; // 保存 acquireFence
}

// frameworks/native/libs/gui/BufferQueueConsumer.cpp
status_t BufferQueueConsumer::acquireBuffer(...) {
    sp<Fence> acquireFence = mSlots[slot].mAcquireFence;
    acquireFence->wait(...); // ← 阻塞等待 GPU 完成渲染
}

1. BufferQueueCore 创建与共享机制

// frameworks/native/services/surfaceflinger/SurfaceFlinger.cpp
status_t SurfaceFlinger::createLayer(...) {
    // 1. 创建唯一的 BufferQueueCore（在 SurfaceFlinger 进程）
    sp<BufferQueueCore> core = new BufferQueueCore();

    // 2. 创建 Producer 和 Consumer，共享同一个 core
    sp<BufferQueueProducer> producer = new BufferQueueProducer(core);
    sp<BufferQueueConsumer> consumer = new BufferQueueConsumer(core);

    // 3. 创建 Layer，持有 Consumer
    sp<Layer> layer = new Layer(..., consumer);

    // 4. 通过 Binder 将 Producer 传递给 App（附带 core 指针）
    outBufferQueueProducer = IGraphicBufferProducer::asBinder(producer);
    return NO_ERROR;
}
✅ 核心机制： BufferQueueCore 仅实例化一次（由 SurfaceFlinger 创建），通过 Binder 传递 IGraphicBufferProducer 接口时，附带 BufferQueueCore 的强引用指针，实现跨进程零拷贝共享。

3. 为什么 Camera 需要 >3 个 buffer？

// Camera HAL3 Pipeline Depth 需求
// ISP 处理延迟 60-100ms > 30fps 帧间隔 33ms
// 必须同时持有多个 request 维持 pipeline 满载

// 最小 buffer 需求公式：
min_buffers = ceil(ISP_latency / frame_interval) + 1

// 30fps (33ms) + 80ms 延迟 → ceil(80/33) + 1 = 4 个
// 60fps (16.67ms) + 80ms 延迟 → ceil(80/16.67) + 1 = 6 个
场景 典型 buffer 数量 原因
普通 UI 2 → 3（Triple） 双缓冲基础，卡顿时动态扩容
Camera Preview 4-8 个 HAL3 pipeline depth 要求（多帧在 ISP 中并行处理）
Video Playback 3-5 个 MediaCodec 输出队列需预分配
分屏/多窗口 6+ 个 每个窗口独立需要 2-3 个 buffer

4. 关键设计哲学

① 生产者-消费者解耦

生产者（App）： 按需生产（有脏区域时）

消费者（SurfaceFlinger）： 持续消费（无论新旧帧）

→ 静态界面实现 零绘制 + 持续显示

② 元数据与像素分离

Binder 传元数据： 低频，~100B/帧（slot ID/时间戳）

Shared Memory 传像素： 高频，~4MB/帧（GraphicBuffer）

→ 最小化 IPC 开销

③ 确定性时序

VSync 全局时钟： 统一调度应用绘制与系统合成

Fence 硬件级同步： 避免 GPU/Display 读写冲突

→ 避免 jank 的根本保障

📌 终极认知： Android 图形系统是 精密的多级流水线，理解 "静态界面靠 Buffer Holding 持续显示" 和 "脏区域驱动按需绘制" 是掌握其低功耗设计的核心。所有机制均在 AOSP 源码中有明确实现，无黑盒逻辑。
