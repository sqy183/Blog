Android 消息机制完全指南 (Handler 机制)

说明： 本文档基于 Android AOSP 源码分析，涵盖概念、源码、设计哲学及高级特性。Mermaid 图表已优化兼容性。

1. 核心概念介绍

Android 的消息机制是 Android 系统进行线程间通信和 UI 刷新的基石。它主要解决了子线程访问 UI 线程以及线程间任务调度的问题。

1.1 四大核心组件

组件 职责 关键特性
Message 消息单元 携带任务数据，内部维护 next 指针用于链表
MessageQueue 消息队列 单链表结构，按时间排序，非线程安全
Looper 消息循环器 每个线程唯一，通过 ThreadLocal 存储，负责轮询
Handler 消息处理器 发送消息入队，接收消息处理，关联特定 Looper

1.2 辅助概念

  • ThreadLocal: 提供线程隔离的变量存储，确保每个线程获取到自己的 Looper。
  • Native Poll (epoll): Linux 内核机制，用于 Looper 空闲时阻塞线程，节省 CPU。
  • Sync Barrier: 同步屏障，用于优先处理异步消息（如 UI 渲染）。

3. 事件流转全过程

消息机制的核心流程分为三个阶段：初始化、发送、轮询处理。

3.2 流程详细解读

  1. 初始化: 主线程启动时自动调用 prepareMainLooper()，创建队列并进入 loop() 死循环。
  2. 发送: Handler 获取当前时间 + 延时 = when，将 Message 插入队列。若插入头部，需唤醒阻塞的 Looper。
  3. 轮询: Looper 不断调用 next()。若无消息，线程在 Native 层阻塞（不耗 CPU）；若有消息且时间到，取出分发。
  4. 处理: 通过 Message 持有的 target (Handler) 回调 handleMessage。
  5. 回收: 消息处理完后，清空数据放回对象池，减少 GC。

4. 源码关键逻辑分析

4.1 消息入队 (enqueueMessage)

MessageQueue 是单链表，插入时需要遍历找到合适的时间位置。

// MessageQueue.java
boolean enqueueMessage(Message msg, long when) {
    synchronized (this) {
        // 1. 检查屏障
        if (mBlocked) { ... }
        
        // 2. 标记消息在使用中
        msg.inUse = true;
        
        // 3. 遍历链表找位置 (按 when 排序)
        Message p = mMessages;
        if (p == null || when == 0 || when < p.when) {
            // 插入头部
            msg.next = p;
            mMessages = msg;
            needWake = mBlocked;
        } else {
            // 插入中间或尾部
            Message prev;
            for (;;) {
                prev = p;
                p = p.next;
                if (p == null || when < p.when) {
                    break;
                }
            }
            msg.next = p;
            prev.next = msg;
        }
        
        // 4. 唤醒
        if (needWake) {
            nativeWake(mPtr);
        }
    }
    return true;
}

4.2 消息轮询 (next)

核心在于 Native 层的阻塞与唤醒。

// MessageQueue.java
Message next() {
    int nextPollTimeoutMillis = 0;
    for (;;) {
        // 1. Native 层阻塞/等待
        nativePollOnce(ptr, nextPollTimeoutMillis);
        
        synchronized (this) {
            // 2. 获取头消息
            Message msg = mMessages;
            if (msg != null) {
                // 3. 检查同步屏障
                if (msg.target == null) { ... } 
                
                // 4. 检查时间
                long now = SystemClock.uptimeMillis();
                if (now < msg.when) {
                    // 时间未到，计算等待时间
                    nextPollTimeoutMillis = (int) Math.min(msg.when - now, Integer.MAX_VALUE);
                } else {
                    // 5. 时间到，取出消息
                    mBlocked = false;
                    if (prev != null) { prev.next = msg.next; }
                    else { mMessages = msg.next; }
                    msg.next = null;
                    return msg;
                }
            } else {
                // 无消息，无限阻塞
                nextPollTimeoutMillis = -1;
            }
            
            // 6. 处理退出
            if (mQuitting) { ... }
        }
    }
}

5. 设计哲学：为什么这样设计？

5.1 为什么 MessageQueue 是单链表而不是 BlockingQueue？

维度 MessageQueue (单链表) BlockingQueue 设计原因
节点对象 Message 本身就是节点 需要额外 Node 包装 内存优化：减少对象创建，配合对象池复用。
排序 按时间 (when) 有序 通常 FIFO 支持延时消息，队头始终是最早执行的任务。
阻塞 Native epoll Java Lock/Condition 性能：Native 层阻塞更高效，唤醒更快。
并发 单消费者 多消费者 UI 线程模型决定只需单线程消费，无需复杂锁竞争。

5.2 为什么 Looper.loop() 不会卡死主线程？

因为 nativePollOnce。当队列无消息时，线程在 Native 层进入休眠状态，释放 CPU 时间片。只有当有新消息入队（触发 nativeWake）或超时，线程才会被唤醒。这是一种事件驱动模型，而非忙等待。

5.3 为什么主线程不需要手动 prepare()？

在 ActivityThread.main() 方法中，系统已经替我们调用了 Looper.prepareMainLooper() 和 Looper.loop()。这是 Android 应用入口的固定模板。

6. 高级特性

6.1 IdleHandler (空闲回调)

当队列中没有消息等待处理时执行。常用于启动优化，将非关键任务延迟到首帧绘制后。

Looper.myQueue().addIdleHandler(new MessageQueue.IdleHandler() {
    @Override
    public boolean queueIdle() {
        // 执行非关键任务
        // 返回 false 表示执行一次后移除，true 表示保留
        return false;
    }
});

6.2 同步屏障 (Synchronization Barrier)

用于确保异步消息（如 UI 渲染）优先于同步消息执行。

  • 原理: 插入一个 target == null 的 Message 作为屏障。
  • 效果: next() 遇到屏障时，跳过所有同步消息，只处理异步消息。
  • 应用: Choreographer 使用此机制保证 VSYNC 信号到来时优先绘制。

6.3 异步消息 (Async Message)

通过 Handler.createAsync(looper) 创建的 Handler 发送的消息标记为异步。可穿越同步屏障。

7. 常见问题 & 解答 (FAQ)

Q1: 如果在 Looper.loop() 之前发送消息，会发生什么？
A: 消息会成功入队并存放在 MessageQueue 中，处于 pending 状态。一旦 loop() 启动，会立即按顺序处理。前提是必须已经调用过 Looper.prepare()。
Q2: sendMessageDelayed 的延时绝对精确吗？
A: 不精确。它保证"至少延迟这么久"。实际执行时间受前方任务耗时、CPU 调度、系统休眠（uptimeMillis 停止计时）影响。
Q3: 如何监控 ANR 或卡顿？
A: 使用 Looper.setMessageLogging(Printer)。在消息分发前后打印日志，计算时间差。若超过阈值（如 200ms），上报堆栈。
Q4: Kotlin 协程 Dispatchers.Main 底层是什么？
A: 底层复用了 Android 的 Handler。协程将任务封装为 Runnable，通过 Handler 发送到主线程 Looper。
Q5: Handler 内存泄漏如何避免？
A: 1. 使用 static 内部类 Handler。
2. 使用 WeakReference 持有 Activity。
3. 在 onDestroy 中调用 removeCallbacksAndMessages(null)。

8. 最佳实践建议

  • 避免耗时操作: 不要在 handleMessage 中进行网络请求或复杂计算，这会阻塞 UI 渲染。
  • 及时清理: 页面销毁时，务必移除 Handler 中的消息，防止内存泄漏和意外回调。
  • 合理使用 IdleHandler: 启动优化时，不要将所有任务塞进一个 IdleHandler，可分批执行。
  • 优先使用协程: 业务逻辑推荐使用 Kotlin Coroutines，底层仍依赖 Handler 但代码更安全简洁。
